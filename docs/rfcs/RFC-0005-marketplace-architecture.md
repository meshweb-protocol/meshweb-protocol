# RFC-0005: Marketplace Architecture & SDK Boundary

**Status:** Proposed (Sprint C Prelude)
**Author:** MeshWeb Core Team
**Created:** 2026-07-20

## Summary
Defines the architectural boundary between the MeshWeb Storage Protocol (Daemon) and the Application Layer Marketplace (SDK). It outlines how Proof-of-Storage generated by the daemon is used by the SDK to negotiate and settle payments.

## Motivation
To strictly maintain the *Payment Verification Invariant* and the *Product Independent* nature of the protocol, the economic layer must operate exclusively outside the core storage engine. We need a clear, formal boundary defining how the daemon and the SDK communicate.

## Specification

### 1. Daemon ↔ SDK Interface
- **Daemon Role:** Operates strictly on `FileID`, `Manifest`, `Erasure Coding`, and `DHT`.
- **SDK Role:** Operates on `User Accounts`, `Vouchers`, `Pricing`, and `Reputation`.
- **Communication:** Local gRPC or REST API exposed by the daemon strictly for the local SDK.

### 2. Proof-of-Storage Flow
- The Protocol V1 daemon already generates cryptographic guarantees (SHA256 hashes of repaired/served shards).
- The daemon exposes a `VerifyShard` local endpoint to the SDK.
- The SDK uses this local verification to determine if a peer has successfully honored a storage deal.

### 3. Voucher Flow (Payment Settlement)
- **Escrow:** Buyers lock fiat-denominated stable value in an escrow contract (handled strictly by the SDK/Smart Contract layer).
- **Voucher Generation:** As the daemon retrieves chunks from a provider, the SDK issues micro-payment "Vouchers" signed by the buyer's private key.
- **Redemption:** The provider's SDK collects these vouchers and submits them to the escrow contract for settlement.

### 4. Pricing Abstraction
- A storage deal's `PricePerGBDay` is negotiated out-of-band by the SDKs *before* the upload (`/meshweb/push/1.0.0`) begins.
- The daemon does not know what a "Price" is. It only pushes data to `peer.ID`s that the SDK explicitly authorizes.

## Open Questions
- How does the SDK inject the authorized list of `peer.ID`s into the daemon's routing table during a storage deal?
- Should the voucher signatures be standard Ed25519 identical to the libp2p node identity, or a separate wallet identity?

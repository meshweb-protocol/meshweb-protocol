# RFC-0007: Proof-of-Storage Flow

**Status:** Proposed (Sprint C)
**Author:** MeshWeb Core Team
**Created:** 2026-07-20

## Summary
Defines the structure and lifecycle of "Proof-of-Storage" challenges. It standardizes how the protocol layer validates that a peer genuinely holds a shard, allowing the SDK layer to release economic vouchers.

## Motivation
The Marketplace needs a cryptographically secure way to verify that a provider is actually honoring their storage deal before paying them. Simply trusting the Health Protocol (`/meshweb/health/1.0.0`) is insecure against Byzantine faults (where a node claims to have a shard but has deleted it). A cryptographic proof is required.

## Specification (Proposed)

### 1. Proof Format
> **Note: This is a placeholder proof mechanism for Sprint C research. The final Proof-of-Storage algorithm may be replaced after rigorous cryptographic evaluation (e.g., Merkle Challenge, Proof of Retrievability (PoR), Provable Data Possession (PDP), etc.).**

- A challenge consists of a `Nonce` generated by the buyer's SDK.
- The provider's daemon must generate a proof by returning:
  `Proof = HMAC(Nonce, ShardData)`
- The buyer's daemon re-calculates the HMAC locally (if they possess the source data or a partial merkle tree) or verifies it against a pre-agreed Merkle Root included in the Manifest.

### 2. Signatures & Identity
- Proofs are signed using the provider's standard libp2p `Ed25519` key to prevent impersonation.
- Vouchers (the payment response to a valid proof) are signed by the buyer's SDK wallet key (which may be distinct from their libp2p identity to allow hardware wallet isolation).

### 3. Verification & Replay Protection
- The SDK stores a record of `(FileID, ShardIndex, Nonce)` to prevent replay attacks.
- Once a proof is validated, the SDK increments the payment sequence and issues the next micro-voucher.

## Drawbacks
Generating an HMAC over a full 64MB shard is computationally expensive and incurs heavy I/O overhead if challenged too frequently. The protocol must implement a Merkle-tree based chunk challenge (e.g., "hash chunk 5, 92, and 14") rather than hashing the entire shard to optimize I/O paths.
